## 2024-05-17 - CPU spin loop due to dynamic timer miscalculation during disconnection
**Learning:** When transitioning a polling loop that tracks state over time (like heartbeat or button repeat intervals) to an event-driven `queue.get(timeout=...)` model, you MUST update the state variables (timers) even when the condition for taking action (e.g., `is_connected()`) is false. If the condition fails, and timers are skipped, the calculated `time_until_next_event` can hit 0. A queue timeout of 0 becomes completely non-blocking, causing a 100% CPU spin loop while disconnected.
**Action:** When calculating dynamic timeouts based on historical timestamps, always ensure a default fallback timeout (e.g., `1.0s`) is applied and state timestamps are reset if the normal processing block is bypassed due to edge cases like disconnections.

## 2024-05-18 - Logging overhead from eager evaluation of expensive functions
**Learning:** Python's `logging` module discards log statements when the log level is higher than the statement's level (e.g. `DEBUG` statement under `INFO` level). However, if the arguments to the log statement involve eager evaluation of expensive operations or function calls like `binascii.hexlify()` inside an f-string, those operations are executed *before* the logging module decides whether to discard the log. This creates significant overhead in hot paths even when logging is effectively disabled.
**Action:** When logging in a high-throughput loop or hot path, especially when the log message involves string formatting of computationally expensive operations (like converting binary data to hex), wrap the logging call in an `if _logger.isEnabledFor(logging.DEBUG):` block. This ensures the expensive operation is only evaluated when the log will actually be emitted.
